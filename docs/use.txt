SECTION 3 — AUTH FLOW PER ROLE (PRODUCTION BEHAVIOUR)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CONSUMER
  Register  →  OTP sent  →  OTP verified  →  Account created (status: approved)
  Login     →  Password match + role match  →  Dashboard access immediately

MANAGER
  Register  →  OTP sent  →  OTP verified  →  Account created (status: pending)
  Login     →  Blocked with "pending admin approval" until admin approves
  Approve   →  Admin logs in → approves manager → manager can now login

ADMIN
  Register  →  OTP sent  →  OTP verified  →  Account created (status: approved)
  Login     →  Password match + role match  →  Dashboard access immediately
  Note: At least one admin must be seeded in MongoDB for the approval flow to
        work — otherwise there is nobody to approve managers.

  1. ADMIN USER — ALREADY SEEDED ✓
     Run `node seedAdmin.js` from the backend directory to create the default
     admin account (safe to run multiple times — skips if already exists).

       Email   : admin@jbvnl.in
       Password: Admin@1234
       Role    : admin  |  Status: approved

     Change the password after first login. The admin account is the only one
     that can approve pending manager accounts.

  2. OTP DELIVERY — CONSOLE LOG (INTENTIONAL)
     OTPs are printed to the server terminal in ALL environments (dev and
     production). This is intentional for this project — no email service is
     needed. When a user registers, check the backend terminal for:

       [OTP] ----------------------------------------
       [OTP] CODE FOR user@email.com: 485921
       [OTP] ----------------------------------------

     When you are ready for a real deployment with actual users, replace the
     console.warn() block in authController.js → sendOtp() with Nodemailer:
       - npm install nodemailer
       - Configure SMTP credentials in .env (SMTP_HOST, SMTP_USER, SMTP_PASS)
       - Send the OTP via transporter.sendMail() instead of console.warn()

  3. REPLACE IN-MEMORY OTP STORE (important before real deployment)
     The otpStore Map is wiped on every server restart. If the backend restarts
     while a user is mid-registration, their OTP is lost and they must start
     over. For a real deployment, store OTPs in a MongoDB collection with a TTL
     index or use Redis.

  4. SECURE JWT_SECRET (important before real deployment)
     Replace "your_super_secret_key_123" in backend/.env with a
     cryptographically random string (minimum 32 characters).
